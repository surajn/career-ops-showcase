Extending an open-source job-search system — the right way

I directed an AI coding agent to extend a popular open-source multi-agent job-search tool with the features my own Director/EM search actually needed — an operational dashboard, better scanner coverage, and a least-privilege email design — without forking it or breaking its upgrade path.

Attribution — what's mine vs. what isn't

The foundation is career-ops (MIT, © Santiago Fernández de Valderrama) — a multi-agent job-search system I did not write. Everything described here is my extension on top of it, built by directing Claude Code and living entirely in the project's own user-extension layer. No base-project code is claimed as mine.

01What I built

  • A durable, zero-token, offline scan dashboard — browse discovered roles by scan date, split into Director+ / Senior-Manager tiers, tagged by Bay-Area vs remote, with search and a manual, persistent status tracker (Interested → Applied → Interviewing → Offer → …).
  • Scanner coverage & filter fixes — repaired a portal whose ATS had migrated (Ashby→Teamtailor), hardened title matching for punctuation-variant leadership titles, widened over-narrow location filters, and added ~34 Bay-Area >$5B AI / SaaS / fintech companies with correct per-vendor ATS detection.
  • A least-privilege email integration — designed a read-only "have I already applied?" check, then made the call to ship a simpler manual status tracker instead.

02Why

The base tool discovers and evaluates roles well. I wanted three things it didn't do for me: an at-a-glance operational view of what surfaced and when; confidence that on-target roles weren't being silently dropped by over-tight filters; and a place to track my own pipeline — all while keeping the tool auto-updatable, so my customizations would survive upstream updates instead of being wiped or causing merge conflicts.

03How — the engineering worth defending

Architecture

Upgrade-safe by design

Studied the project's data-contract and its updater internals, then placed 100% of my code in gitignored user-layer paths the updater's USER_PATHS guard refuses to touch — with zero imports from system code. It survives update-system.mjs and git pull, and upstream changes can't break it.

Diagnosis

Root-cause, not hacks

Traced why real roles were dropped — US-CA-Menlo Park matched no location keyword; Director/Sr Director - Software Engineering matched no title keyword — weighed three fixes (co-occurrence matcher vs. punctuation normalization vs. keyword enumeration) and chose the one that was correct and upgrade-safe.

Security

Least-privilege email

Designed email access as read-only from the start: Microsoft Graph Mail.ReadBasic (no bodies), OAuth device-code (no password ever stored), network egress allow-listed to Microsoft only, and revocable in one click.

Judgment

Ship value, drop friction

When the email OAuth setup hit a real wall, I pivoted to a manual, persistent status tracker (browser storage + an exportable JSON file) that delivered the outcome with zero external dependencies and no ongoing cost.

04Impact

  • Recovered roles that were being silently filtered out — Snowflake / Meta / Netflix leadership roles via the location fix; a live Salesforce Director role via the title fix.
  • Broadened coverage by ~34 high-value companies; scans now surface hundreds of roles across dates in one browsable, status-tracked view.
  • A reusable, fork-free pattern for extending an open-source tool that keeps taking upstream updates — the transferable lesson.

Demo above runs on a sanitized ~24-role public sample; the real build runs against my live scans.

05How I worked with the AI agent

This was built across several sessions by directing an AI coding agent — codebase exploration, weighing design trade-offs, and enforcing hard constraints (upgrade-safety, least-privilege, zero token cost), including course-correcting when a chosen path hit a wall. The artifact is real; the skill on display is technical direction of AI under real-world constraints.
